# CS-230-Operating-Platforms
The Gaming Room was a client looking to expand their existing game, Draw It or Lose It, so it could run across multiple platforms instead of being limited to a single environment. The application needed to support multiple teams playing at the same time, which meant it had to handle concurrent users, distributed communication, and secure data management. Based on those requirements, I designed a client-server system hosted on a Linux server, using RESTful communication and a relational database to manage users, teams, and game sessions. The goal was to create a scalable and secure architecture that could support growth over time.

One thing I believe I did well in the design document was clearly connecting the client’s requirements to specific technical decisions. For example, recommending a Linux-based server made sense because of its stability and cost-effectiveness. Using REST for distributed communication allowed the system to remain platform-independent, since clients could connect through standard HTTP methods. I also made sure to separate concerns between storage, memory, networking, and security rather than treating them as one large topic. That helped keep the design organized and easier to understand.

Working through the design document before writing code was helpful because it forced me to think about the system as a whole. Instead of jumping straight into implementation, I had to consider how different components would interact. Thinking about distributed systems and REST architecture early made it clear that scalability and stateless communication were important. Addressing security, such as HTTPS, role-based access control, and hashed passwords, during the design phase also made it easier to understand how protection should be layered rather than added later.

If I could revise one part of the document, I would strengthen the technical justification behind some of the architectural decisions. While I explained why I chose a Linux server, REST communication, and a relational database, I could improve the document by adding clearer explanations of trade-offs and potential limitations. For example, I could expand on how REST supports stateless communication and scalability in distributed systems, or include more discussion about how the system would handle increased user load over time. I could also improve the non-functional requirements section by being more specific about performance expectations and system availability. Adding that level of detail would make the design feel more complete and show a deeper understanding of how architectural decisions impact real-world performance.

Interpreting the user’s needs required translating general goals like “expand to multiple platforms” and “support multiple teams” into architectural decisions. That meant choosing a web-based client-server model, scalable infrastructure, and a structured database. Considering the user’s needs is important because the design directly affects performance, usability, and long-term maintainability. If those needs are not clearly defined at the beginning, technical decisions may not align with what the client actually wants.

When approaching software design, I focused on breaking the system into logical layers: client interface, server logic, database storage, and network communication. I considered scalability, cost, and security as part of the design instead of treating them as separate concerns. In future projects, I would continue using structured design documents to map requirements to architecture before coding. Planning at this level made the development process more intentional and helped me understand how large systems are organized.
